## きっかけ：New Chatなのになぜ知っている？

Claude Codeのようなエージェント型CLIを自作しているなかで、ふと気になることがあった。

**New Chatを開いたはずなのに、ClaudeはわたしがLangGraphでAI Agent CLIを作っていることを知っている。**

これはバグでも魔法でもなく、きちんと設計された仕組みだ。この記事ではその構造を解説する。

---

## メモリの階層構造

AI Agentのメモリは大きく3層に分かれる。

```
┌─────────────────────────────────────┐
│  External Memory（永続化）           │
│  - ファイル / DB / ベクトルDB        │
├─────────────────────────────────────┤
│  In-Context Memory（セッション内）    │
│  - 会話履歴（トークン上限まで）       │
├─────────────────────────────────────┤
│  In-Weights Memory（学習済み知識）    │
│  - LLM自体が持つ知識                 │
└─────────────────────────────────────┘
```

### セッション内（New Chatで消える）

- 会話履歴全体（`messages[]`配列）
- ツール実行結果
- ファイルの読み込みキャッシュ

### New Chatをまたいで残る（External Memory）

Claude Codeは `CLAUDE.md` というファイルをプロジェクトルートに置く方式を採用している。

```
プロジェクト/
├── CLAUDE.md          ← プロジェクト固有の記憶
├── ~/.claude/
│   └── CLAUDE.md      ← ユーザ全体のグローバル記憶
```

書かれる内容の例：
- プロジェクトの技術スタック・規約
- よく使うコマンド
- 「このファイルは触るな」などの制約
- ユーザーの好み（テスト方針、コードスタイル等）

---

## 「何を覚えるか / 忘れていいか」の判断軸

| 判断軸 | 覚える対象 | 忘れていい対象 |
|--------|-----------|---------------|
| **再利用性** | プロジェクト構造、技術選定理由 | 一時的なデバッグ過程 |
| **変化頻度** | アーキテクチャ方針（低頻度） | 現在のエラー内容（高頻度） |
| **コスト** | 「また聞くのが面倒」な情報 | LLMが自分で推論できること |
| **副作用リスク** | 「やってはいけないこと」 | 「やったこと」の詳細ログ |

## Claude.aiのuserMemoriesはまさにこの設計

このUIの「メモリ」機能はExternal Memoryの実装例だ。仕組みはシンプル：

```
過去のセッション
    ↓ (システムが重要情報を抽出・保存)
外部DB (userMemories)
    ↓ (New Chat開始時にsystem promptへ注入)
今のセッション ← Claudeが「知っている」理由
```

実際にわたしのuserMemoriesには以下が保存されている：
- 職業・居住地などの属性
- 進行中プロジェクトの状態（agent-ai-cli: LangGraph ReActまで完了）
- GitHubオーガニゼーション名・ブログURL
- 業務内容・過去の学習履歴

逆に**忘れられている情報**：
- 過去の会話の具体的なやりとり
- デバッグ中に試したコードの詳細
- 解決済みエラーの内容

**要約・抽象化された情報だけが残る**、というのが実際の動作だ。

## メモリ更新のタイミングとトリガー

Anthropicの内部実装は非公開だが、観察と一般的な設計パターンから推測できる。

**リアルタイム更新（発言ごと）は考えにくい** — コストと遅延の観点から。

最も自然な設計は**非同期・バッチ処理**：

```
セッション終了 or 一定時間経過
    ↓
バックグラウンドでLLMが会話を分析
    ↓
「記憶に値する情報」を抽出・既存メモリと統合
    ↓
DBを更新
```

| 書く | 書かない |
|------|---------|
| 職業・居住地などの属性 | 一時的な感情・気分 |
| 進行中プロジェクトの状態 | 解決済みエラーの詳細 |
| 明示的な好み・制約 | 雑談の内容 |
| 長期的に有効な情報 | セッション固有のコンテキスト |

## 自作CLIへの応用

LangGraphのReActパターンで実装済みであれば、こう整理できる：

```python
class AgentMemory:
    # Layer 1: In-context（自動）
    conversation_history: list[Message]  # LangGraph State

    # Layer 2: External（要実装）
    project_memory: str    # CLAUDE.md相当 → ファイル読み込み
    user_memory: str       # ~/.agentcli/memory.md 相当

    # Layer 3: Episodic（高度）
    past_sessions: VectorDB  # 過去セッションの要約を検索
```

セッション終了時にLLMに要約・取捨選択させるプロンプト例：

```python
SUMMARIZE_PROMPT = """
この会話を振り返り、次のセッションに持ち越すべき情報だけを抽出してください。

判断基準:
- 残す: プロジェクト固有の知識、ユーザーの好み、制約事項
- 捨てる: 解決済みの一時エラー、試行錯誤の過程、雑談

出力: 箇条書きで簡潔に（200字以内）
"""

async def update_memory_on_session_end(conversation: list[Message]):
    summary = await llm.invoke(SUMMARIZE_PROMPT, conversation)
    memory_store.merge(summary)  # 単純上書きではなくmerge
```

**mergeが重要**：単純上書きだと過去の情報が消える。LLMに「既存メモリと矛盾しないか確認してから更新」させるのがポイント。

---

## まとめ

- AI Agentのメモリは **In-Weights / In-Context / External** の3層構造
- New Chatをまたぐには **External Memory** が必須
- Claude Codeは `CLAUDE.md` ファイル、Claude.aiは `userMemories` がその実装
- 更新は**セッション終了後の非同期バッチ**が自然な設計
- 忘れる判断は**再利用性・変化頻度・コスト・副作用リスク**の4軸
- 自作CLIでは**セッション終了フック＋LLMによる要約＋mergeロジック**で実現できる
